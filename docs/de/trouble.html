dotProject Problembehandlung

Einstellungen in der includes/config.php

  1. Wurde im Zuge einer Neuinstallation von dotProject eine Kopie der Datei config-dist.php mit dem Namen config.phperstellt?
  2. Wurden bei einer etwaigen Aktualisierung eventuell neue Konfigurationseinstellungen der neuen config-dist.php in die vorhandene Datei config.phpübernommen bzw. wurden die alten Einträge korrekt übernommen?
  3. Ist die Konfigurationsvariable root_dir korrekt gesetzt?
  4. Befinden sich nach dem schließenden PHP Tag ?> Leerstellen (White Spaces) oder leere Zeilen(Carriage Returns)? Falls ja, müssen diese alle gelöscht werden. Sonst werden die HTMl-Kopfdaten zu früh gesendet.
  5. Wurde bei einer eventuellen Aktualisierung die Konfigurationsvariable check_legacy_password auf true gesetzt? Falls nicht, hole man dies nach. Dies erlaubt die Konvertierung alter Datenbankstrukturen für die Passwortabfrage. 
    $dPconfig['check_legacy_password'] = true;

Dateieinstellungen in der php.ini

  1. register_globals = off
    Normalerweise sollte dotProject zufriedenstellend mit "off" arbeiten. Man kann aber auch mal versuchen, diese Variable auf "on" zu setzen.
  2. session.auto_start = 0
    Im Allgemeinen sollte dies auf "0" gesetzt sein. Ist die Variable auf "1" gesetzt und es treten Probleme auf, sollte man es einmal mit "0" versuchen.
  3. session.save_path = /tmp (*nix style)
    session.save_path = "d:/apache/php/sessiontemp" (Windows-Stil)
    Man überprüfe die Existenz dieses obligatorischen Verzeichnisses.

PHP

  1. Es wird PHP Version 4.1.x oder höher benötigt.
  2. Ist MySQL auf einem *nix Server korrekt in PHP einkompiliert? Dies kann überprüft werden, indem man ein kleines phpSkript mit folgendem Inhalt schreibt und ausführt:: 
    <?php phpinfo(); ?>
    Man führe diese Datei (über den WebServer) im Browser aus und suche nach Konfigurationseinträge für MySQL.
  3. Wenn die Gantt-Charts nicht funktionieren: Wurde die GD-Bibliothek korrekt in PHP einkompiliert?

Was man noch versuchen kann

In der Datei ./index.php befindet sich eine Zeile mit etwa folgendem Inhalt: 

// If you experience a 'white screen of death' or other problems,
// uncomment the following line of code:
//error_reporting( E_ALL );
Man lösche die Kommentare ("//") am Anfang der error_reporting-Zeile und beobachte, ob noch immer Fehlermeldungen auftreten.

Allgemeines

Der Fortschrittsstatus für Projekte wird über ein gewichtetes Mittel der Dauer der assoziierten Aufgaben berechnet. Deshalb kann es vorkommen, dass der Fortschritt eines Projektes geringer als 50% ist, obwohl die beiden assoziierten Aufgaben einen Fortschritt von 50% besitzen. Dies ist darauf zurückzuführen, dass die Aufgaben von unterschiedlicher Dauer sind. Hat keine Aufgabe eine positive Dauer, wird der Fortschritt für das projekt immer 0% betragen.

Möchte man das Modul zur Übersetzungsverwaltung benutzen, so müsssen das Verzeichnis locales/{country code} und alle beinhalteten Dateien (von der Form *.inc) unter einem POSIX basierten Betriebssystem die entsprechenden Rechte besitzen, so dass der WebServer über Schreib-&Lesezugriff verfügen kann.

Hilfe!

Wenn alles andere keinen Erfolg bringt, sende man sein Problem an das SourceForge-Forum oder den SF-Tracker von dotProject (http://sourceforge.net/projects/dotproject/). Für eine umfassende Fehleranalyse sollten auf jeden Fall folgende Informationen der Problemschilderung beigefügt werden:

  • Die Version von PHP und MySQL.
  • Name und Version des Webservers.
  • Name und Version des WebBrowsers.
  • Die benutzte Version von dotProject (offizielles Release oder CVS).

Man beschreibe das Problem so gut als möglich. Je mehr Informationen bereit gestellt werden, desto größer ist die Wahrscheinlichkeit, dass dem Problem Abhilfe geschaffen werden kann. Man versuche beispielsweise nicht zu sagen: "Die Kontaktlinks funktionieren nicht". Beschreibungen von der Form "Wenn ich auf die Buchstaben der Kontakte-Startseite klicke, lande ich in der Benutzerliste des Benutzerverwalters." sind wesentlich hilfreicher und effektiver.

Ein anderes Beispiel könnte sein: "Wenn ich versuche, ein Projekt zu bearbeiten, bekomme ich die Fehlermeldung 'Bitte zuerst eine Firma erstellen'!". Die entscheidende Information fehlt hier, nämlich dass der Benutzer lediglich Leserechte für das Projektmodul besitzt.
